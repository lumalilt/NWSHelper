# Security Policy

## Reporting a vulnerability

Do not open a public GitHub issue for a suspected security vulnerability.

Use one of these paths instead:

1. GitHub private vulnerability reporting for this repository, if it is enabled.
2. If private reporting is not available, contact the repository owner privately through GitHub and include `security` in the subject or first line.

Please include:

- a clear description of the issue
- affected release version or commit
- reproduction steps or proof-of-concept details
- impact assessment if known
- any suggested mitigation or patch direction

## Disclosure expectations

- Please allow time for triage and remediation before public disclosure.
- Avoid posting exploit details, secrets, license keys, or signing material in public issues or discussions.

## Scope notes

- Public repo issues can include packaging, release automation, updater metadata, GUI/CLI behavior, and other public-source concerns.
- Store credentials, signing certificates, and other protected release secrets must never be included in reports or logs.
